Deep Dives

The six infrastructure layers underneath your agent and which ones won't last

A detailed breakdown of the agent stack surfaced today, and it's the clearest map of this space I've seen. Here's what actually matters for builders:

The six layers, rated:

  1. Compute/sandboxing Production-ready. E2B (Firecracker microVMs, ~$32M raised) and Daytona ($24M Series A, 90ms cold starts) represent two diverging bets: ephemeral versus persistent sandboxes. Both survive because different workloads need different models. Durability: high.

  2. Identity/communication The current answer is "give the agent an email address." AgentMail ($6M seed, General Catalyst) lets you programmatically create real inboxes for agents. It works because email is universal, not because it's right. Native A2A protocols are coming. Build on this now, but don't mistake it for a permanent commitment. Durability: medium (2-3 year runway).

  3. Memory/state Mem0 is the leader: $24M raised, 41K GitHub stars, 14M downloads, AWS exclusive for their Agent SDK. Their LOCOMO benchmark results: 26% better accuracy than OpenAI's built-in memory, 91% lower latency, 90% fewer tokens. The platform risk is real if labs absorb memory the way ChatGPT absorbed search, standalone memory companies get squeezed. Durability: uncertain.

  4. Tool access/integration Composio ($29M, Lightspeed Series A) handles auth, OAuth, and pre-built connectors for 250+ SaaS apps. The NM integration problem (every agent managing credentials for every tool independently) is unsustainable at enterprise scale. This layer survives until MCP goes truly universal which is years away. Durability: high near-term.

  5. Provisioning/billing Stripe Projects launched in late March. It tokenizes payment credentials into scoped tokens for agent-initiated transactions, so an agent can provision a Neon database (ready in 350ms) without your raw card details leaving Stripe's vault. What's still missing: agent-to-agent payments, metered billing that matches agent compute patterns, and financial observability across multi-agent workflows. Durability: high for Stripe.

  6. Orchestration This is the layer that matters most and exists least. Gartner logged a 1,445% surge in multi-agent inquiries between Q1 2024 and Q2 2025. Deloitte estimates the autonomous agent market could be 15-30% larger if enterprises orchestrated agents better. LangChain/CrewAI/AutoGen are framework-level tools, not infrastructure. The gap between "three agents in a notebook" and "50 agents across enterprise systems with failure recovery, cost controls, and audit logs" is enormous. Nobody has built the Kubernetes for agents yet. Durability: whoever builds this wins.

The math that should change your architecture decisions: Five services at 99% uptime each = 95% combined uptime. At 97% each, you're at 86%. Every primitive you stack compounds in the wrong direction.

Builder takeaways: Context engineering beats prompt engineering (what you feed the agent matters more than how you word it). Eval-driven development replaces unit tests your feedback loop is success rate and cost per task. And stack literacy knowing which layers to build on, which to build, and which to watch from a safe distance is now the core skill.

Anthropic blocks Claude subscriptions for third-party agents community migrates to API

Anthropic now blocks Claude subscription plans from being used in third-party agent harnesses, plus first-party harness use in some configurations. Third-party apps draw from extra usage capacity, not plan limits. The community response is essentially unanimous: bring your own API key.

The cost reality this surfaces is significant. Ryan Carson published concrete numbers: running a genuinely useful always-on agent on Opus 4.6 API costs $100-200/day roughly $3-6K/month. That's the real price of moving past consumer subscriptions to production agent usage. PerceptualPeak claims to have reverse-engineered a CLI bypass after 10 hours, but the more durable response is AlexFinn's: "If you have a local model running on your Mac mini, no corporation will ever be able to ban you." The dependency on a single lab's pricing decisions is now a stated business risk, not just an inconvenience.

  • Karpathy burned $200 on X's API in 30 minutes of hacking. His verdict: good direction for Read endpoints, "really excessive" for Write, docs poorly structured for agent ingestion because they're spread across individual pages instead of consolidated. API access API affordability.

  • Dotta (Paperclip creator) on agent infrastructure independence: "You will have thousands of agent employees and your business cannot be at the whims of one lab." The argument for neutral, open-source orchestration layers is getting sharper.

  • Claude Code multi-repo workflow worth implementing: Clone all related repos into a single VS Code workspace at the parent directory level. A 16-line pull-all script keeps them current. Lets Claude Code trace cross-service interactions that single-repo queries miss. Non-engineers on customer-facing teams can query the codebase directly, reducing engineer interruptions to near-zero.

  • Two Claude Code features most users skip: /compact compresses long sessions into a tight summary without losing key context use it before your context window degrades, not after. Global CLAUDE.md lets you set machine-wide preferences ("never ask clarifying questions, make a decision") that apply across all projects.

  • Ryan Carson, deep in agent territory, still says: "You still need to know how to code." Agents are force multipliers. Understanding what they're building remains the moat.
