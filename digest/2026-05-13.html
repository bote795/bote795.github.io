Agent Architecture: Two Frameworks Worth Stopping For

Your agent is rediscovering 85% of its context every run.

Classic RAG worked for question answering. It breaks for agents. The problem isn't retrieval method. Agents must assemble context before acting: current account record, user permissions, controlling policy, the right document section, the table behind a metric, the prior decision that governs the next step. When the system doesn't prepare that bundle, the model improvises. Wrong refunds get issued. Stale policies get cited. Outdated metrics land in board decks, confidently wrong.

Vector search is getting demoted from "the architecture" to one component inside a broader knowledge layer. That layer includes retrieval, but also document structure, semantic data models, access control, provenance, memory, and write-back. Pinecone's Nexus claims roughly 85% of agent effort goes to knowledge retrieval treat that as vendor framing, but the failure mode is real and well-documented in production.

The seven-question test for whether your agent knowledge layer can handle production:

  1. What is the work object? Name it: customer, contract, filing, metric, code change. An abstract search query is not a work object.
  2. What's the right retrieval unit? Chunk, section, table, record, or graph neighborhood. Wrong unit = model wastes effort compensating.
  3. Which source is authoritative? Relevant and controlling are different things. A Slack thread about a policy is not the policy.
  4. What permissions apply? Permission-aware before the model sees content, not cleaned up after.
  5. What provenance has to be preserved? Source trails for decisions, finance, legal, compliance, code reviews.
  6. What context should be compiled ahead of time? Project briefs, customer packets, repo conventions don't make the agent rediscover these every run.
  7. What should the agent write back? Label writes as observed, inferred, user-confirmed, or stale. Never let model-generated memory silently become instruction.

The action item isn't "replace your vector database." Write the retrieval contract: what exact bundle does your agent need before it can act reliably? "Relevant context" is too vague. Name the actual fields. For a support refund agent that means: customer plan, region, product version, purchase history, refund policy, threshold, prior exceptions, current ticket, approved response language. For a code review agent: issue, diff, touched files, tests, failing logs, ownership, repo conventions, recent related bugs.

Don't ask your model to rebuild the business every time it runs.

Agentic commerce broke into six layers. Most products cover two.

The old checkout button hid six distinct commercial responsibilities behind one human click. Agents expose every layer, and if you haven't named who owns each one, the rows labeled "nobody" are your failure modes.

The six layers:

  1. Discovery: Where intent forms. ChatGPT, Gemini, browser agents.
  2. Authorization: Structured proof the agent was allowed to act. This outlasts the transaction. It is not the same as payment.
  3. Payment credential: The scoped token or wallet. Never a raw card handed to a bot. Stripe's Link, Visa Intelligent Commerce, Mastercard Agent Pay, PayPal agentic services are all targeting this.
  4. Settlement: Cards for consumer retail. Stablecoins (Coinbase x402, Stripe Machine Payments Protocol) for software paying software. Different risk models, different rails.
  5. Merchant relationship: Who owns the customer, order, data, returns, support path.
  6. Governance: Policy, budgets, approval thresholds, audit trails. This is the layer AWS is building toward with Amazon Bedrock AgentCore Payments.

OpenAI and Stripe launched Instant Checkout inside ChatGPT in September 2025. By March 2026 it was scaled back — Walmart's head of AI told Wired that conversion inside the chatbot ran roughly three times lower than the same items requiring a click-out (per CNBC's coverage of the revamp). What survived was the Agentic Commerce Protocol (ACP) itself — the spec for passing structured purchase data between agent and merchant. The checkout location was a product question. The protocol is the durable piece.

AWS's AgentCore Payments isn't strategic because it owns the payment rail. It's strategic because the agent runtime sees the work around the payment: which tools ran, which policy was active, what the agent did before and after. That's the enterprise governance control point.

Builder action: Map discovery authorization payment credential settlement merchant relationship governance. Name who owns each. Any row that says "nobody" is a future incident.

Supply Chain Warning

Source: Silicon Valley Eats the War — From the Trenches, citing voennoedelo.com, dronexl.co, Pravda Ukraine, Pew Research, and the author's earlier "AI Silicon Tax" reporting.

AI data centers doubled the cost of Ukraine's fiber-guided drones. Drone-grade fiber optic cable jumped from roughly $4-5/km to $34-50/km by spring 2026 (per voennoedelo.com and dronexl.co coverage compiled in the linked Substack). A Starlink terminal now costs less than a single 35km fiber spool. China controls roughly 60% of global fiber production, 70% of rare earths, and 90% of rare earth processing — feeding both hyperscaler GPU clusters and both sides of the war from the same supply chain. The same pattern runs across RAM (prices up ~187% year-over-year per the "AI Silicon Tax" analysis), mature-node chips, copper, and grid electricity (up ~267% near Virginia data center hubs, per Pew Research wholesale-price data). Most AI infrastructure builders aren't modeling commodity constraints as operational risk. They should be.

Watch

Nick from Orgo lays out a complete playbook for running a one-person AI agent business charging $5K-$10K/month per customer. The offer is a "digital employee" — unlimited agents, unlimited usage, full management — sold as a productized service to executives in legacy industries (law, insurance, marketing, real estate, wholesale, manufacturing). The episode covers the offer construction, target market selection, fulfillment stack (Hermes, Codex, Orgo, Composio, Agent Mail, Obsidian, GPT-5.5), and the meta-pattern of using agents to set up other agents.

Key takeaways:

  • Sell a digital employee, not an agent: The customer should never touch tokens, models, or infrastructure. They get a named employee (e.g., "Mia") with her own email. Mindset shift: Before: priced/marketed as "agent with X credits/month" → After: priced as "unlimited agents, unlimited usage, $5K flat" — because pricing on tokens kills the magic and creates anxiety.
  • Unlimited works because customers don't actually need much: They think they need 5-10 agents; reality is 1-3 well-built agents handle 90% of the value. Promise unlimited; deliver focused. This is how you control costs while removing friction.
  • Avoid healthcare/finance, target legacy people-businesses: Best industries are marketing agencies, law firms, insurance, manufacturers, wholesalers, real estate. Reasons: high people-headcount = high waste, they want to be AI-native but don't know how, and low regulatory burden compared to healthcare/finance.
  • Niche after you diverge, not before: Contrarian to standard "pick a niche first" advice. Try 3-5 industries broadly, see which one pulls you, then go subniche (geography + vertical, e.g., "commercial real estate agencies in Florida"). Design thinking principle: diverge then converge.
  • Sell to executives, not departments: Decision-makers across every industry share the same pain — too many emails, meetings, follow-ups, open loops. Build a generic executive template, then layer vertical-specific skills on top (e.g., demand letters for matrimonial law).
