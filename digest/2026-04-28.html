GPT-5.5 scored 87 on a 23-deliverable executive work package where Opus 4.7 scored 67. The same week, an AI agent wiped a production database and all backups in 9 seconds. Capability is outrunning guardrails. The builders who route models deliberately and scope agent permissions tightly will eat the ones who don't.

The Execution Gap Is Measurable

Source: Nate B. Jones, "ChatGPT 5.5 scored 87 where the next best model scored 67" — three production-grade tests (Dingo executive package, Splash Bros data migration, Artemis II 3D viz) run head-to-head across GPT-5.5, Opus 4.7, Sonnet 4.7, and Gemini 3.1 Pro.

Real-world testing on three production-grade tasks puts GPT-5.5 at 87.3 vs Opus 4.7 at 67.0, Sonnet 4.7 at 65.0, and Gemini 3.1 Pro at 49.8. The test: a 23-deliverable executive package. Word docs, a 17-slide PowerPoint with 26 embedded media files, spreadsheets with real formulas, an interactive dashboard, and legal research backed by 34 source URLs. Gemini's "Office files" were HTML wearing wrong extensions. The gap comes from pre-training depth, not inference tricks, so it compounds across every task.

GPT-5.5 is the first frontier model to pass a data-migration canary check. Given 465 messy business files (CSVs in three schemas, corrupted JSON, scanned handwritten receipts), it rejected planted fake records that Opus and GPT-5.4 both absorbed as real data. The fakes: Mickey Mouse, Test Customer, and a $25K bogus payment. It merged all 7 planted duplicate customer pairs and caught all 13 name-typo orders. Backend hygiene (enum normalization, orphan handling, schema constraints) still needs human review before production.

Visual taste remains Opus territory. A 3D Artemis II visualization showed better lighting, composition, and visual authority from Opus 4.7. GPT-5.5 won on information density. For anything that needs to look beautiful from a blank canvas, start with Claude.

Model routing cheat sheet:

  • Complex multi-step execution (files, code, tools, data, artifacts) GPT-5.5
  • Blank-canvas visual taste (UI from scratch, 3D, design) Opus 4.7
  • UI with a reference mockup Generate with Images 2, implement with GPT-5.5 in Codex
  • Architecture and planning Opus 4.7, then hand execution to GPT-5.5
  • Data migrations GPT-5.5 for first pass, human for final validation
  • Long-form writing GPT-5.5 holds argument structure better than any prior OpenAI model

What changes in practice:

  1. The system matters as much as the weights. Codex computer use + Images 2 + GPT-5.5 form an integrated stack. GPT-5.5 trapped in a chat window is dramatically underused.
  2. For data work, require source provenance, rejected records, conflict tables, normalized enums, and reconciliation reports in your prompt. Never let the model self-certify production data.
  3. Define your artifact contract up front. Tell the model what files, formats, and deliverables you expect. "Real usable artifacts" is part of the assignment.
  4. The frontier advantage is no longer answering hard questions. It's carrying work across deliverables, formats, and constraints without losing coherence.

AlexFinn corroborates from daily use: "Claude Code has been dethroned. Codex w/ ChatGPT 5.5 is the smartest model, best all around AI app." His underrated observation: Codex auto-writes test scripts and runs them in a built-in browser.

Agents Are Faster Than Your Guardrails

The PocketOS incident is now the canonical case study for agent permission scoping. An AI agent deleted a production database and all backups in 9 seconds. Root cause: the agent found keys with permissions across multiple environments and used them, believing it was doing the right thing. A separate Cursor + Railway incident killed a B2B rental SaaS in 30 hours — Cursor's agent and Railway's API combined to nuke production, and Ryan Carson's takeaway sums it up: "an industry that markets AI safety faster than it ships it."

Both incidents share the same root cause: agents granted permissions whose blast radius outran the operator's mental model. The fix isn't a smarter agent — it's tighter scopes (per-environment keys, no production access from a dev session) and verified backups. Carson's response was to schedule a Devin cron job that tests backup restores monthly, so a 9-second wipe doesn't become a 30-hour business-killer. If you're shipping agents that touch production data, copy that pattern this week.

Videos

Stop using Claude. Start using Codex? — @gregisenberg

Watch on YouTube — Greg Isenberg interviews Riley Brown on why he switched from Claude to OpenAI Codex as his daily driver. Riley's pitch: Codex is the "super app" — a single GUI that combines vibe coding, knowledge work (docs/sheets/slides), browser use, computer use, and scheduled automations on GPT-5.5. The kicker: you can still run Claude Code inside Codex via Cmd-J terminal, so switching doesn't mean abandoning Claude — it means hosting both in one surface.

5 takeaways:

  1. The "super app" thesis is converging UX, not just a model bet. Codex, Claude desktop, and Cursor are all settling on the same three-pane layout (project folders/threads left, agent middle, live artifact right). The choice is moving from "pick a model" to "pick an interface that hosts every model."
  2. Run Claude Code inside Codex. Cmd-J opens a terminal, type claude, set --dangerously-skip-permissions. You get Claude's design quality + Codex's GUI + your existing Claude subscription tokens.
  3. Skills > plugins. Plugins require OpenAI approval (Slack, Notion, Canva, Remotion, Expo). Skills are just folders with a skill.md instruction file — you describe a repetitive task, the AI scaffolds the skill for you.
  4. Automations are made after a successful run, not before. Run the task once, refine the output, then tell the agent "make this happen every Friday at 9am." Codex creates the recurring job from the working chat.
  5. Examples beat instructions. "One good example is amazing, five good examples and it does great every time." Highest-leverage move for a company today: collect and organize good output examples in a Notion DB the agent can reference.

Inside AppLovin's Profit Engine — @20VC

Watch on YouTube — Harry Stebbings interviews Adam Foroughi (CEO, AppLovin) on rebuilding from the 2022 92% stock collapse to a $150B+ market cap with 84% EBITDA margins and ~400 people in the core business. Most candid CEO interview on running lean at scale this year.

5 takeaways:

  1. Token budgets are the new hiring quotas — don't set them. Setting AI token budgets creates the same dysfunction as headcount targets: people optimize for usage metrics, not outcomes. The right frame: can you attribute token spend to revenue? If yes, invest aggressively. If not, you're creating slop.
  2. Engineers as product managers. AppLovin has no product org. Engineers own the KPIs and the product decisions. In the AI-native era, what matters is imagination + ability to audit/review code — not just writing it. Engineers who can't get 10–100x output from AI tools are a liability.
  3. Cut headcount before the tools are fully mature. AppLovin cut 40–50% of staff during a year of triple-digit growth, targeting roles headed toward automation. Restructuring early means adoption isn't held hostage by people protecting their jobs.
  4. You can't get lean by firing 50% — fire 99% and rebuild. If mediocrity has set in, A players are likely already gone. Buyouts and reorgs leave mediocrity intact. The only real fix is a hard reset.
  5. Cash flow minus SBC is the only real metric. $1B in cash with $1B in equity granted = zero real value created. AppLovin grants ~$300M/year in SBC against a $150B market cap — one of the lowest rates in tech.
